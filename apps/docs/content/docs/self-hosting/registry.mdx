canette ships with an optional in-cluster container registry (`registry:2`) enabled by default. This page explains how it works, how credentials are managed, and how to set up TLS for local development clusters where Let's Encrypt isn't an option.

## How the registry works

When `registry.enabled=true` (the default), the Helm chart sets up:

- A `registry:2` deployment in the `canette-system` namespace, listening on port 5000
- An `HTTPRoute` that exposes it at `registry.{domain}` through your Gateway
- Auto-generated credentials stored in the `canette-registry-credentials` Secret, reused on every `helm upgrade`
- A docker config Secret (`canette-registry-auth`) mounted into every build job automatically

There are two distinct paths for registry traffic:

| Who | Address | Protocol |
|-----|---------|----------|
| BuildKit (push from build jobs) | `registry.canette-system.svc.cluster.local:5000` | HTTP — internal ClusterIP, no TLS |
| Kubelets (pull to run app pods) | `registry.{domain}` | HTTPS — through your Gateway |

BuildKit pushes over plain HTTP to the internal ClusterIP address. Kubelets pull over HTTPS through the Gateway, so **TLS termination on the Gateway is required** for image pulls to work.

## Credentials

Registry credentials are auto-generated on first install and reused on every upgrade — no configuration needed. Build jobs receive the credentials automatically via the `canette-registry-auth` docker config Secret.

To rotate credentials or set your own, see the [Helm reference](/docs/self-hosting/helm#registry-authentication).

## TLS for local development (mkcert)

On a local cluster (k3s, kind, etc.) without a public IP, [Let's Encrypt](/docs/getting-started/installation#1-gateway-api-with-tls-termination) won't work. [mkcert](https://github.com/FiloSottile/mkcert) generates locally-trusted certificates backed by a CA you install yourself.

<Callout type="info">
  This is more involved than cert-manager. You need to install the mkcert CA in your cluster, on every node that runs workloads, and on any client machine that pushes images. Budget 15–20 minutes.
</Callout>

### Step 1 — Generate a certificate

```bash
brew install mkcert
mkcert -install   # installs the local CA into the macOS keychain

mkcert "registry.apps.example.com"
# Produces: registry.apps.example.com.pem  and  registry.apps.example.com-key.pem
```

### Step 2 — Create the TLS Secret in Kubernetes

```bash
kubectl create secret tls registry-tls \
  --namespace canette-system \
  --cert="registry.apps.example.com.pem" \
  --key="registry.apps.example.com-key.pem"
```

### Step 3 — Configure your Gateway to use the cert

How you reference this Secret depends on your Gateway implementation. With Traefik, reference it in your `Gateway` resource or annotate the `HTTPRoute`. Consult your Gateway implementation's docs for the exact approach.

### Step 4 — Install the mkcert CA on each node

Every node that runs app pods needs to trust the mkcert CA so that kubelets can pull images from `registry.{domain}`.

```bash
# Find the CA file
mkcert -CAROOT
# e.g. /Users/you/Library/Application Support/mkcert

# Copy to each node (Ubuntu/Debian)
scp "$(mkcert -CAROOT)/rootCA.pem" root@<NODE_IP>:/usr/local/share/ca-certificates/mkcert-rootCA.crt
ssh root@<NODE_IP> update-ca-certificates
```

On RHEL/CentOS nodes: copy to `/etc/pki/ca-trust/source/anchors/` and run `update-ca-trust extract`.

For k3s, also add the CA to `/etc/rancher/k3s/registries.yaml` on each node:

```yaml
configs:
  "registry.apps.example.com":
    tls:
      ca_file: "/usr/local/share/ca-certificates/mkcert-rootCA.crt"
```

Then restart k3s: `systemctl restart k3s`.

## Using an external registry instead

To disable the in-cluster registry and push to an external one (GHCR, ECR, Docker Hub, etc.):

```yaml
registry:
  enabled: false

externalRegistry:
  username: "<username>"
  password: "<password>"

controller:
  pullRepo: "ghcr.io/myorg/"
```

See the [Helm reference](/docs/self-hosting/helm#infrastructure-optional) for the full set of external registry values.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| `x509: certificate signed by unknown authority` on Mac | mkcert CA not in keychain | Run `mkcert -install` |
| `x509: certificate signed by unknown authority` on node | mkcert CA not installed | Copy `rootCA.pem` and run `update-ca-certificates` |
| `ImagePullBackOff` on app pods | Node doesn't trust the CA or DNS doesn't resolve | Check CA install, verify `registry.{domain}` resolves to Gateway IP from the node |
| `401 Unauthorized` on push | Build job credentials out of sync | Run `helm upgrade --reuse-values` to regenerate the docker config Secret |
| Registry HTTPRoute not resolving | Gateway not configured for that hostname | Check your Gateway's listener configuration covers `registry.{domain}` |
